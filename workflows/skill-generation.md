# Skill 生成流程

本文件定义了 agent 何时应该生成 skill 草稿、如何生成、以及什么情况下不应该生成。仅在触发时加载。

---

## 核心原则

**你已经非常聪明。只记录你凭训练数据不知道的东西。**

在决定是否生成 skill 前，对每条信息做以下检查：

* 这个知识是否存在于公开文档中且容易找到？如果是，不记录。
* 未来遇到同样问题的 agent，是否能凭自身能力解决？如果是，不记录。
* 这条信息值不值得占用未来每次检索的 token 预算？如果不值得，不记录。

Skill 库是一个有限的共享资源。每新增一条，所有 agent 未来检索时的噪音就多一分。只有真正有复用价值的非平凡经验才值得写入。

---

## 触发判断

当一次问题解决结束后，评估是否需要生成 skill。

### 应该记录的信号（满足任一即可）

* 解决方案违反直觉，或与官方文档的描述不一致
* 需要阅读源码、翻 issue tracker 或查阅非官方渠道才找到的方案
* 问题表象和根因之间存在非显而易见的关联（例如 OOM 的根因是 checkpoint 保存策略）
* 涉及特定版本的行为差异、隐式默认值、或未文档化的 API 行为
* 解决过程中踩了容易重复踩的坑
* 用户主动说"记录一下"、"这个值得记住"或类似表述

### 不应该记录的情况（满足任一即排除）

* 简单的 typo、拼写错误、语法错误
* 查文档就能直接找到答案的配置项（如"某参数的默认值是多少"）
* 一次性的环境问题（如"我的 pip 版本太旧"），除非该环境问题具有普遍性
* 问题本身是由用户的临时实验代码引起的，不会出现在正式代码中
* 解决方案是显而易见的（如"没装这个包，装上就好了"）
* Agent 凭训练数据就能正确处理的问题

### 判断后的行为

* **判断为应该记录** ：告知用户你打算生成一个 skill 草稿，简述要记录什么，然后执行生成
* **判断为不应该记录** ：不做任何动作，不需要告知用户
* **不确定** ：简短地问用户"这个问题的解决方案要记录为 skill 吗？"，不要长篇解释

---

## 生成前检查

生成草稿前，先执行以下检查：

1. **查重** ：读取 `~/.agent-memory/skills/index.yaml`，检查是否已有覆盖相同问题的 approved skill

* 如果已有完全匹配的 skill → 不生成，告知用户已有记录
* 如果已有部分相关的 skill → 在草稿的 frontmatter 中标注 `related: [skill-id]`，审核时可能需要合并

1. **粒度把控** ：一个 skill 只解决一个问题。如果一次 debug 过程中涉及多个独立知识点，拆成多个草稿

## Skill 草稿模板

```markdown
---
title: "简明标题，描述问题而非解决方案"
tags: [tag1, tag2, tag3]
created: YYYY-MM-DD
context: "哪个项目/什么场景下遇到的"
status: draft
related: []          # 可选，index.yaml 中已有的相关 skill id
---

### 问题

[一句话描述问题现象。包含具体的错误信息或可观测行为，使未来检索时能快速匹配。]

### 根因

[解释因果链。不要只说"X 导致 Y"，要说清楚为什么 X 会导致 Y。]

### 解决方案

[具体怎么解决的。附可执行的代码片段或命令。]

### 踩坑点

[容易忽略的细节、边界条件、与直觉相悖的行为。如果没有则省略此节。]
```

---

## 模板填写规范

### title

title 是 index.yaml 中的主要匹配字段。写得差就永远不会被检索到。

* 描述问题而非方案（`DeepSpeed Stage3 checkpoint 保存不完整` ✓，`使用 torch.save 替代 deepspeed.save_checkpoint` ✗）
* 包含关键技术名词，便于 index 检索时匹配
* 控制在 20 字以内

### tags

tags 是辅助匹配字段，覆盖 title 未能涵盖的检索维度。

* 3-5 个标签，覆盖：技术栈（如 `deepspeed`）、问题类别（如 `checkpoint`）、场景（如 `distributed-training`）
* 使用小写英文，用连字符连接多词标签
* 不要用过于泛化的标签（如 `python`、`bug`、`error`）

### 问题

这一节是未来检索的第一匹配面。写作时假设另一个 agent 正在搜索同样的问题——它会用什么关键词？把这些词写进去。

包含具体的错误信息或可观测的异常行为。

### 根因

Skill 最核心的价值在这里——连接表象和本质。

* 解释因果链，不要只说"因为 X 导致了 Y"，要说清楚为什么 X 会导致 Y
* 如果涉及版本特定行为，注明版本号

### 解决方案

精简到可以直接复制执行的程度。对每行代码问自己：未来的 agent 能否凭训练数据推断出这行？如果能，删掉。只保留非显而易见的部分。

* 给出可直接执行的步骤或代码
* 如果有多种方案，只记录推荐方案，不要列出所有尝试过的错误路径

---

## 文件命名与存储

* 文件名格式：`YYYY-MM-DD-简短描述.md`（例：`2026-02-28-stage3-save-issue.md`）
* 存入 `~/.agent-memory/skills/drafts/` 目录
* 简短描述用小写英文，用连字符连接，控制在 5 个词以内

---

## 生成后行为

1. 告知用户草稿已生成，给出文件路径
2. 输出草稿的 title 和一句话摘要，供用户快速判断
3. 提示用户可以通过"review skills"触发审核流程
4. 回到正常工作流，不要中断当前任务
