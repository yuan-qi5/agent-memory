# Skill 审核流程

本文件定义了协助用户高效审核 skill 草稿的完整流程。Agent 负责预处理和执行，用户负责最终决策。仅在用户触发审核时加载。

---

## 触发方式

用户说"review skills"或类似表述时，读取本文件并按以下流程执行。

---

## 阶段一：草稿审核

这是审核的主要流程，处理 `~/.agent-memory/skills/drafts/` 中的所有待审核草稿。

### 步骤 1：扫描

读取 `~/.agent-memory/skills/drafts/` 目录，列出所有 `.md` 文件。

* 如果没有待审核草稿，告知用户"没有待审核的 skill 草稿"，然后跳到阶段二
* 如果有草稿，告知用户数量和时间范围（如"共 3 个草稿，最早的来自 2026-02-15"）

### 步骤 2：预处理检查

对每个草稿执行以下检查。这些检查的目的是降低用户的审核成本——让用户看到的是带有建议的摘要，而不是需要从头阅读的原文。

#### 2.1 去重检查

将草稿的 title、tags、问题描述与 `~/.agent-memory/skills/index.yaml` 中已有的 approved skill 对比：

* **完全重复** ：问题和方案与已有 skill 基本一致 → 建议丢弃
* **部分重叠** ：同一问题但有新的信息或不同角度 → 建议合并到已有 skill
* **无重复** ：继续后续检查

#### 2.2 可检索性检查

title 和 tags 决定了这个 skill 未来能否被其他 agent 找到。这是最容易被忽略但影响最大的质量维度。

检查项：

* **title 是否描述了问题而非方案** ：未来的 agent 搜索时是带着问题来的，不是带着答案来的
* **title 是否包含关键技术名词** ：如果 title 中没有技术栈名称（如 `deepspeed`、`nccl`），检索时可能匹配不到
* **tags 是否覆盖了合理的检索维度** ：至少覆盖技术栈、问题类别、场景三个维度
* **tags 是否过于泛化** ：`python`、`bug`、`error` 这类标签没有区分度，不应出现
* **问题一节是否包含了其他 agent 会搜索的关键词** ：具体的错误信息、异常行为描述

如果发现问题，在审核报告中给出具体的修改建议（如"建议将 title 从'训练中断'改为'DeepSpeed Stage3 checkpoint 保存导致训练中断'"）。

#### 2.3 内容完整性检查

* **问题** ：是否有具体的错误信息或可观测行为？仅说"XX 不工作"不够
* **根因** ：是否解释了因果链？仅说"因为 X 导致 Y"而不解释为什么 X 会导致 Y 不够
* **解决方案** ：是否具体到可以直接执行？是否包含可复制的代码或命令？
* **踩坑点** ：不是必填项，但如果方案中有容易忽略的细节，应该有这一节

#### 2.4 版本与过期风险检查

扫描草稿内容，标注以下风险：

* **显式版本依赖** ：方案中提到了具体版本号（如 `deepspeed==0.12.3`、`PyTorch 2.1`）→ 标注"⚠️ 依赖特定版本，版本升级后可能失效"
* **隐式版本依赖** ：方案依赖某个 API 或行为但未注明版本 → 建议在草稿中补充版本信息
* **时效性内容** ：方案引用了特定的 GitHub issue、PR、或 workaround，这些可能在未来被官方修复 → 标注"⚠️ 可能在未来版本中被修复"

不要尝试验证版本是否仍然是最新的（除非当前环境支持联网查询）。只需要标注出存在版本依赖这个事实，提醒用户关注。

#### 2.5 分类建议

根据草稿的 tags 和内容，建议放入 `~/.agent-memory/skills/approved/` 下的哪个子目录：

* 如果已有匹配的子目录（如 `deepspeed/`、`pytorch/`）→ 建议放入对应目录
* 如果不匹配任何已有子目录但内容明确属于某个领域 → 建议创建新子目录（仅当该领域已有或预计会有 3 个以上 skill 时）
* 如果暂时无法归类 → 建议放入 `general/`

同时建议语义化的文件名（从 `2026-02-28-stage3-save-issue.md` 改为 `stage3-checkpoint.md`）。

### 步骤 3：呈现审核报告

对每个草稿，输出以下格式的审核报告：

```
### 📝 [草稿标题]
文件：drafts/2026-02-28-xxx.md
摘要：[一句话描述这个 skill 记录了什么]

检查结果：
- 去重：✅ 无重复 / ⚠️ 与 [已有 skill id] 部分重叠 / ❌ 与 [已有 skill id] 重复
- 可检索性：✅ title 和 tags 清晰 / ⚠️ [具体问题和修改建议]
- 完整性：✅ 各字段完整 / ⚠️ [缺少什么]
- 版本风险：✅ 无版本依赖 / ⚠️ [具体的版本依赖说明]
- 分类建议：→ approved/[子目录]/[建议文件名].md

💡 建议：通过 / 修改后通过 / 建议丢弃 / 建议合并到 [已有 skill]
理由：[一句话说明为什么给出这个建议]
```

如果草稿数量较多（>5 个），先输出一个总览表格，再逐个展开：

```
| # | 标题 | 建议 | 主要问题 |
|---|------|------|----------|
| 1 | DeepSpeed Stage3 checkpoint... | ✅ 通过 | 无 |
| 2 | NCCL timeout 排查 | ⚠️ 修改后通过 | title 缺少技术名词 |
| 3 | pip 安装失败 | ❌ 丢弃 | 显而易见的方案 |
```

### 步骤 4：执行用户决策

逐个或批量获取用户对每个草稿的决策，然后执行：

#### 通过

1. 根据审核中的修改建议（如果有），修改草稿内容
2. 将 frontmatter 中的 `status` 从 `draft` 改为 `approved`
3. 将文件从 `skills/drafts/` 移动到 `skills/approved/[子目录]/[语义化文件名].md`
4. 在 `skills/index.yaml` 中新增条目（id、title、tags、path）
5. 在 `skills/stats.yaml` 中新增条目（use_count: 0）
6. 如果目标子目录的 `_summary.md` 需要更新，一并更新
7. 删除 `drafts/` 中的原文件

#### 修改后通过

1. 按用户指示修改草稿内容（可能包括：改写 title/tags、补充根因、修正方案等）
2. 将修改后的内容展示给用户确认
3. 确认后，执行与"通过"相同的步骤 2-7

#### 丢弃

1. 删除 `drafts/` 中的草稿文件
2. 告知用户已删除

#### 合并到已有 skill

1. 读取目标 approved skill 的内容
2. 将草稿中的新信息整合到已有 skill 中（补充新的踩坑点、更新解决方案、增加新的上下文等）
3. 将合并后的内容展示给用户确认
4. 确认后，更新 approved skill 文件
5. 删除 `drafts/` 中的草稿文件
6. 如果合并导致已有 skill 超过 100 行，建议拆分

---

## 阶段二：库健康检查（可选）

草稿审核完成后，对 `approved/` 中的现有 skill 进行健康度评估。如果用户只想审核草稿不需要健康检查，可以跳过。

### 使用频率分析

读取 `~/.agent-memory/skills/stats.yaml`，标注以下情况：

* **零引用 skill** ：自创建以来 `use_count` 始终为 0 → 可能从未被需要，建议评估是否值得保留
* **长期未使用 skill** ：`last_used` 超过 3 个月 → 可能已过时或不再相关，建议评估
* **高频 skill** ：`use_count` 排名前列 → 这些 skill 值得投入精力持续维护和完善

注意：stats.yaml 中的数据是 best-effort 的近似值，不是精确统计。零引用不一定意味着没有价值（可能是因为某些 agent 环境不支持更新统计），但值得用户关注。

### 版本过期扫描

遍历 `approved/` 下所有 skill，检查内容中的版本号依赖：

* 如果方案依赖特定版本且该版本距今已超过 6 个月，标注为"建议验证版本兼容性"
* 如果当前环境支持联网查询，可以尝试检查最新版本并对比

### 规模与组织检查

* 单个 skill 文件是否超过 100 行？→ 建议拆分
* 某个子目录下的 skill 数量是否过多（>10 个）？→ 建议细分子目录
* `general/` 下是否有可以归入特定子目录的 skill？→ 建议迁移
* 子目录总数是否超过 10 个？→ 建议合并相近的分类

### 健康报告格式

```
## 📊 Skill 库健康报告

当前状态：
- approved skill 总数：N 个
- 子目录数：N 个
- 最近一次使用：YYYY-MM-DD

需要关注：
- ⚠️ [N] 个 skill 从未被引用：[列出 title]
- ⚠️ [N] 个 skill 超过 3 个月未使用：[列出 title]
- ⚠️ [N] 个 skill 存在版本过期风险：[列出 title 和涉及的版本]
- ⚠️ [N] 个 skill 超过 100 行，建议拆分

高频 skill（值得持续维护）：
- [title]（引用 N 次，最近使用 YYYY-MM-DD）
```

---

## 注意事项

* 审核流程中 agent 的角色是 **预处理和建议** ，最终决策权始终在用户手中
* 不要在没有用户确认的情况下执行任何文件操作（移动、删除、修改 approved skill）
* 如果 drafts/ 中的草稿数量很大（>10 个），建议分批审核而不是一次处理全部
* 审核完成后，确认所有文件操作都已执行（index.yaml 已更新、drafts/ 中的已处理文件已清理）
